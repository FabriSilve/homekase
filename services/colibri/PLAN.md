# Colibrí Service for Homekase

## Overview

Add [Colibrí](https://github.com/JustVugg/colibri) as a homekase service — a lightweight C engine running GLM-5.2 (744B MoE, int4) on CPU with streaming expert loading from disk. Provides an OpenAI-compatible API for opencode and other coding tools.

## Hardware context

- **Server**: ASRock DeskMini X300, AMD Ryzen 7 5700G (8c/16t), 26.8GB RAM
- **Disks**: Kingston SNVS500G (NVMe, 465GB), ST1000LM048 (HDD, 1TB)
- **GPU**: AMD Radeon Vega (integrated) — not supported by Colibrí, CPU-only
- **Current state**: NVMe fully partitioned, model will initially live on HDD (`/storage`)
- **Future**: Dedicated NVMe for model + repos (`/code`)

## Architecture

Follow the **assistant pattern** — compose file and Dockerfile live in `services/colibri/`, installer writes config to `/opt/homekase/colibri/`.

### Files to create

| File | Purpose |
|------|---------|
| `services/colibri/Dockerfile` | Build Colibrí from source (debian + gcc + single C file) |
| `services/colibri/docker-compose.yml` | Single container: engine + OpenAI-compatible API |
| `lib/services/colibri.sh` | Installer: `deploy_colibri()`, `remove_colibri()`, `_update_colibri()` |

### Files to modify

| File | Change |
|------|--------|
| `lib/services/service.sh` | Add `"colibri:Local LLM inference (GLM-5.2 744B MoE)"` to `SERVICES` array |

## Installer flow (`homekase add colibri`)

1. `require_root` + `header "Installing Colibrí LLM Engine"`
2. **Port** — `port_wizard "colibri" 1` (default: next available)
3. **Model path** — `ask_input "Model directory" "/storage/colibri/model"`
   - Detect disk type via `/sys/block/.../queue/rotational`
   - If HDD (`rotational=1`): warn about slower cold starts, ask confirmation
   - Check free disk space ≥ 380GB, warn if insufficient
4. **Projects path** — `ask_input "Projects directory" "/storage/colibri/projects"`
5. **RAM budget** — `ask_input "RAM budget (GB)" "16"`
6. **Context length** — `ask_input "Context window (tokens)" "32768"`
   - Hint: "max 131072, higher = more RAM for expert cache"
7. **API key** — auto-generate via `openssl rand -hex 32`, show to user
8. **Tailscale** — `tailscale_serve_setup`
9. Write `.env` + `docker-compose.yml` to `/opt/homekase/colibri/`
10. `mkdir -p` for model and projects dirs
11. `docker compose build` (compile C engine, ~30s)
12. `docker compose up -d`
13. Wait for health check (`GET /health`)
14. `config_app_set colibri installed true` + port + tailscale + paths
15. Print success message with URL, API key, and opencode config instructions

## Docker Compose

```yaml
services:
  colibri:
    build: .
    container_name: colibri
    restart: unless-stopped
    ports:
      - "${BIND_ADDR}${PORT}:8000"
    volumes:
      - ${MODEL_DIR}:/model
      - ${PROJECTS_DIR}:/projects:ro
    environment:
      - COLI_MODEL=/model
      - COLI_API_KEY=${API_KEY}
      - RAM_GB=${RAM_GB}
      - CTX=${CTX}
      - KV_SLOTS=1
      - PIPE=1
      - COLI_COLOR=0
    deploy:
      resources:
        limits:
          memory: ${RAM_LIMIT}
          cpus: "${CPU_LIMIT}"
    labels:
      com.homekase.service: colibri
      com.homekase.port: "${PORT}"
      com.homekase.tailscale: "${TS}"
```

### Conditional env vars (set in .env based on disk detection)

- `DIRECT=1` only if NVMe detected (O_DIRECT for sustained NVMe reads)
- `COLI_NUMA` skipped (single-socket CPU)

## Environment variables in .env

```bash
PORT=4090
BIND_ADDR=
MODEL_DIR=/storage/colibri/model
PROJECTS_DIR=/storage/colibri/projects
API_KEY=<auto-generated>
RAM_GB=16
CTX=32768
RAM_LIMIT=18g       # RAM_GB + 2GB headroom for OS
CPU_LIMIT=12        # Leave 4 cores for other services
TS=false
```

## Context length rationale

GLM-5.2 uses compressed MLA attention: 576 floats/token (2.3KB) instead of 32,768. KV cache sizing:

| Context | KV cache per slot | With 16GB RAM |
|---------|------------------|---------------|
| 16,384 | ~37MB | trivial |
| 32,768 | ~75MB | trivial |
| 65,536 | ~150MB | fine |
| 131,072 (max) | ~300MB | fine |

Main RAM consumers: dense weights (~9.9GB) + expert streaming cache (~6GB). KV context is negligible.

## API key rationale

1. Many clients (opencode, aider) refuse to connect without an API key
2. Security when exposed via Tailscale or LAN

## Remove flow (`homekase remove colibri`)

1. Remove Tailscale mapping
2. `docker compose down` (keeps model files by default)
3. Prompt: "Delete model directory (~370GB)? [y/N]"
4. Prompt: "Delete projects directory? [y/N]"
5. `config_app_remove colibri`

## Update flow (`homekase update colibri`)

1. `stop_service "colibri"`
2. Rebuild Docker image (C compilation, ~30s)
3. `docker compose up -d`
4. Model files are untouched, only the engine binary updates

## RAM behavior

- Running: 16GB allocated via Docker memory limit
- Paused/stopped: all memory released immediately
- Resume: model reloads in ~32s

## Disk I/O considerations

- Colibrí is read-only on model files (no writes, no fragmentation)
- HDD: ~150 MB/s sequential, cold starts slower but learning cache warms over time
- NVMe: ~2-3 GB/s, `DIRECT=1` for optimal performance
- Background tasks over days spread I/O load thin
- No risk of breaking disk I/O for other services
